## Summary

bnxt (Broadcom NetXtreme-C/E) is the driver for Broadcom's high-end server NIC silicon: BCM5731x/5741x (10/25G NetXtreme-C, Cumulus + Thunderbolt), BCM5750x (Thor 100/200G, deployed at scale in Azure HC-series + bare-metal cloud), BCM5760x (Thor2 200/400G), BCM5770x (Pegasus). The driver also covers BlueRoc/SmartNIC variants. This is one of the most-deployed datacenter NICs after mlx5, especially in Microsoft Azure HBv4/HC-series HPC + bare-metal-cloud deployments and on-prem datacenters.

bnxt is structurally distinct from mlx5/ENA/gVNIC/MANA in one key way: it uses **HWRM** (Hardware Resource Manager) — a versioned, auto-generated firmware ABI spanning ~10000 lines of TLV-style request/response definitions in `include/linux/bnxt/hsi.h`. Every driver↔firmware operation is an HWRM request with a typed opcode + request struct → typed response struct. The HSI header is generated by Broadcom from the firmware spec; backward-compatibility shims (FW-version-conditional features) are pervasive.

Layered architecture:

- **`bnxt_hwrm`** (~820 lines) — HWRM transport. DMA-mapped request/response buffers, "short input" optimization (small reqs sent inline via DMA-bypass), async events through a separate event ring, sentinel-based memory-safety check on request pointers.
- **`bnxt`** (~17500 lines) — PCI lifecycle + netdev + NAPI + ring management + MSI-X + NPAR (NIC Partitioning, multi-PF) + multifunc (multiple netdevs per function). The monolithic core.
- **`bnxt_sriov`** (~1400 lines) — SR-IOV. PF allocates resources for VFs via HWRM (`HWRM_FUNC_VF_CFG`), per-VF MAC/VLAN/spoofchk/trust/rate.
- **`bnxt_vfr`** (~600 lines) — VF representor for switchdev mode.
- **`bnxt_tc`** (~2100 lines) — TC HW-offload. Maps tc-flower rules to CFA (Communications Fabric Architecture) flow table entries. Supports ct (conntrack), tunnel encap, decap.
- **`bnxt_xdp`** — XDP integration.
- **`bnxt_ulp`** (~625 lines) — Upper Layer Protocol aux-driver registration. The bnxt_re RDMA driver (separate kmod) registers as a ULP and shares HWRM + EQ resources.
- **`bnxt_devlink`** (~1330 lines) — devlink integration: params, regions (NVM, register dump, live-fw-trace), health reporters (FW reset, RoCE).
- **`bnxt_ptp`** — PHC + per-packet timestamp.
- **`bnxt_dcb`** — DCB/PFC/ETS for QoS in lossless RoCE networks.
- **`bnxt_ethtool`** (~5800 lines) — massive ethtool surface: stats, RSS, coalesce, ring, link, fec, NVM access, RX-flow-rule install.
- **`bnxt_coredump`** — HW crash dump capture to userspace.

Total ~34000 lines covered (the largest single driver Rookery has documented so far, comparable to mlx5e+mlx5_core combined).

## Rust translation posture

bnxt is the highest-line-count single-driver target. The translation strategy must be modular:

- **HWRM as the central typed-API.** Replace the ~10K-line `hsi.h` macros + accessor functions with a `bnxt-hsi` crate of typed structs (`bilge` for bit-packed fields, generated from the upstream HSI header via build.rs). Each HWRM request is a method on `Hwrm`: `hwrm.func_qcaps(opts) -> Result<FuncQcapsResp, HwrmError>`. The `__hwrm_req_init` / `hwrm_req_send` / `hwrm_req_drop` lifecycle becomes a RAII guard (`HwrmReq<'_, ReqType>`) that auto-frees DMA buffers on drop.
- **PCI lifecycle phase typing.** `BnxtPriv<Probed → HwrmReady → ResCfgQueried → Online>` — methods that need HWRM are gated on `HwrmReady`; ring create gated on `ResCfgQueried`.
- **Ring abstraction.** `RxRing` / `TxRing` / `CqRing` / `NqRing` (NQ = Notification Queue, the EQ-equivalent on Thor+) as distinct types. The two ring topologies (Wh+/SR/Thor — different ring hierarchies) become trait-distinguished: `Topology5704` vs `TopologyThor` with a common `Ring` trait.
- **SR-IOV.** Per-VF state machine like mlx5 eswitch's Vport; `Vf<Disabled → Configured → Enabled<Mode>>`.
- **TC HW-offload.** CFA flow rule install pipeline mirrors mlx5's TC pipeline — typed `parse_tc_actions → CfaRule → HWRM_CFA_FLOW_*` lowering.
- **NPAR / multifunc** — each partition (NPAR creates up to 16 PFs per port) is a separate `BnxtPriv`; shared resource arbitration through devlink params.
- **bnxt_ulp aux-bus.** RoCE driver (bnxt_re) attaches via the ulp registration. Rookify with typed aux-bus identical to mana's pattern.
- **HWRM async events** — typed enum dispatch.

The grsec/PaX section is mandatory: bnxt is heavily used in SR-IOV + cloud deployments, has had numerous CVEs (CVE-2024-35814 nested-NIC overflow, CVE-2021-3635 ethtool OOB, multiple HWRM-handling bugs), and the sheer HSI ABI surface (10K lines) makes parser hardening structurally important.

## Compatibility contract

REQ-1: PCI ID table — all NetXtreme-C/E silicon: BCM5731x (Cumulus), BCM5741x (Thunderbolt), BCM5750x (Thor, the high-volume 100G chip in cloud), BCM5760x (Thor2, 200/400G), BCM5770x (Pegasus). VF IDs covered. Frozen.

REQ-2: HWRM ABI — `include/linux/bnxt/hsi.h` is the firmware ABI. Driver-FW negotiated version: each request struct has an HWRM version it was introduced in; driver checks against firmware-reported HWRM_VER_GET version before issuing. Frozen against baseline; new HWRM versions tracked as future PRs.

REQ-3: HWRM transport — 16KB DMA-mapped req/resp buffer per CPU; short-input opt for requests <128B. Sentinel check on every send (defends against use-after-free of the req buffer).

REQ-4: Async event queue — separate ring, separate vector; events for link state, RESET_NOTIFY, error recovery, ULP-targeted events (RoCE QP errors), VF events.

REQ-5: NQ (Notification Queue) vs CP (Completion Processor) ring topology — Thor and later use NQ-on-top-of-CP, where the NQ is the IRQ source and CPs are the per-direction completion rings. Older silicon uses a flat CP-only model. Driver supports both via topology selection at probe.

REQ-6: VNIC abstraction — a VNIC is the binding between a set of RX rings + an RSS context + a MAC filter list. Each netdev has at least one VNIC; multi-VNIC is used for stateful offload (TC HW-offload secondary VNICs). VNICs are hardware-owned objects allocated via HWRM_VNIC_ALLOC.

REQ-7: SR-IOV — full PF/VF: PF can enable up to 256 VFs (silicon-dependent); per-VF resource caps allocated through HWRM. Per-VF: MAC (`HWRM_FUNC_CFG`), VLAN, spoofchk, trust, max BW. VFs use a separate driver (bnxt_vf) but the binary is the same.

REQ-8: Switchdev — VF representor netdevs (`bnxt_vfr.c`); TC offload via CFA. Mode switch through devlink: `devlink dev eswitch set ... mode switchdev`.

REQ-9: TC HW-offload — tc-flower with `mirred {redirect, egress}`, `vlan`, `tunnel_key`, `pedit`, `csum`, `gact`, `ct`, `sample`, `mpls`, `police`. Maps to CFA flow table install via `HWRM_CFA_FLOW_*`.

REQ-10: ULP (Upper Layer Protocol) — bnxt_re (RDMA) registers as a ULP; shares the bnxt EQ vector for QP completions; allocates QPs through HWRM_RING_ALLOC + HWRM_QUEUE_ALLOC.

REQ-11: NPAR (NIC Partitioning) — multi-PF per port: up to 16 PFs sharing one physical port, each with its own MAC/VLAN/BW partition. Configured at FW level; driver supports per-PF as if it were independent NIC.

REQ-12: Devlink params — `bnxt_devlink.c` params for: `enable_remote_dev_reset`, `gre_ver_check`, etc. + firmware regions (NVM, register dump, FW trace).

REQ-13: PTP — PHC clock + per-packet TX/RX timestamps (when supported by silicon).

REQ-14: DCB — PFC + ETS + 802.1Qaz CEE+IEEE modes. Critical for RoCE lossless networks.

REQ-15: HW crash dump — on FW crash, `bnxt_coredump.c` collects HW state into userspace blob via devlink-region snapshot.

REQ-16: FW reset — RESET_NOTIFY async event triggers driver-initiated quiesce → wait-for-FW-ready → re-init.

## Architecture

**Monolithic core, modular peripherals.** `bnxt.c` is the single largest file in upstream Linux at ~17500 lines. It is the "kernel" of the driver: PCI lifecycle, netdev ops, NAPI poll loop, ring lifecycle, MSI-X allocation. Peripherals (sriov, vfr, tc, xdp, ulp, devlink, ptp, dcb, ethtool, coredump, gso, hwmon, dim) sit in separate files but call into bnxt.c heavily for ring access. Rookery's translation must respect this: `Bnxt` is the central type, peripherals are typed extensions (`impl Bnxt`).

**HWRM transport.** A per-CPU DMA region for short_input (sub-128B reqs), a shared DMA region for full reqs and resps. Each request has:
1. `__hwrm_req_init` — allocate request slot, set `seq_id`, set up DMA mapping.
2. Caller fills request struct.
3. `hwrm_req_send` — write request to DMA region, ring HWRM doorbell, wait for response signal (polling or interrupt).
4. Caller reads response struct.
5. `hwrm_req_drop` — free request slot, unmap DMA.

The sentinel check (`hwrm_calc_sentinel`) detects a request struct being used after being freed — historically a productive bug source. Rookery's `HwrmReq<'_, T>` RAII guard eliminates the use-after-free class structurally (the request is unusable after the guard drops).

**Ring topology dispatch.** Older Wh+/SR (5731x/5741x) has one ring level — completion ring per direction, NAPI polls them. Thor (5750x) introduced a two-level: NQ (Notification Queue) sits on top of CP (Completion Processor) rings. The NAPI vector fires on the NQ, the NQ has per-CP entries that say "go poll CP_n". This indirection lets one MSI-X vector cover many CP rings for high queue counts. Driver detects the topology at probe and configures NAPI accordingly. Rookery uses a `Topology` trait with `Topology5704` and `TopologyThor` impls.

**VNIC abstraction.** A VNIC is a HW object that owns a set of RX rings + an RSS context (indir table + hash key + hash type) + a MAC filter list. Multiple VNICs per function let TC HW-offload route specific flows to specific VNICs. Default VNIC is created at netdev open; additional VNICs allocated by `HWRM_VNIC_ALLOC` and freed by `HWRM_VNIC_FREE`. Per-VNIC RSS context allows independent RSS configs.

**SR-IOV.** PF reserves resources for VFs via `HWRM_FUNC_VF_RESOURCE_CFG` (queues, MSI-X, VNICs, RSS contexts, flow rules). VF probe runs the same bnxt binary; the driver detects VF-mode via PCI cap and shapes itself accordingly (no PF-only ops, fewer HWRM commands accessible, restrictions on devlink). Per-VF config (MAC, VLAN, trust, etc.) is set by PF via `HWRM_FUNC_CFG` targeted at the VF's function ID.

**Switchdev + VF representor.** `bnxt_vfr.c` creates representor netdevs in switchdev mode. Each VFR has its own slow-path send path that wraps packets in metadata-bearing send descriptors with source-VF tagging. TC rules on representors install CFA flows in HW.

**TC HW-offload.** Maps tc-flower match-and-action to CFA (Communications Fabric Architecture) flow table entries. CFA supports up to N flow entries per function (silicon-dependent). The `bnxt_tc.c` pipeline: parse flower spec → resolve actions → call HWRM_CFA_FLOW_ALLOC / _FREE. Conntrack offload integrates with the kernel's CT framework: HWRM_CFA_CT_*. Tunnel encap/decap (VXLAN, Geneve, GRE) supported.

**ULP aux-bus.** `bnxt_ulp.c` registers a ULP slot. The bnxt_re RDMA driver matches on this slot and calls `bnxt_ulp_register` to install RDMA-specific HWRM filters and reserve resources. Bnxt_re shares the HWRM transport, EQ vector for QP completions, and the PCI MSI-X pool. Rookify with typed aux-bus.

**Devlink + health reporters.** bnxt has multiple devlink-health reporters: fw, fw_fatal, fw_activate, etc. Each reporter tracks a specific failure mode; FW reset is driven through these. Devlink regions for NVM access, register dump, live FW trace.

**Crash dump.** `bnxt_coredump.c` walks the FW's coredump segment table and produces a userspace-consumable binary blob. Triggered manually via devlink or automatically on FW reset.

## Hardening

- HWRM request slot leak detection via the `bnxt_hwrm_ctx` registry; remove path asserts all slots freed.
- HWRM response timeout: 25s default; past timeout, driver enters FW-reset path.
- Sentinel check on every send.
- Per-VNIC RSS context bounds-checked against silicon caps.
- TC HW-offload action set bounded against CFA cap (max actions per flow).
- SR-IOV VF resource alloc validated against PF resource pool; over-commit refused.
- Async event payload validated against expected type before dispatch.
- Per-ring stats use 64-bit atomic counters (no overflow on long-lived rings).
- Coredump segment iteration bounded.

## Grsecurity/PaX-style Reinforcement

This driver inherits the standard PaX/Grsecurity surface and reinforces it with:

- **PAX_USERCOPY** — ethtool ioctl ingress (~80 ops), devlink-region read, NVM access all through bounded `copy_from_user`.
- **PAX_KERNEXEC** — `bnxt_netdev_ops`, `bnxt_ethtool_ops`, HWRM request/response dispatch, ULP ops, devlink-health reporter ops, `pci_error_handlers` all `const` after init in W^X memory.
- **PAX_RANDKSTACK** — ethtool / netlink / sysfs / dcb / ptp ioctl entry paths are per-syscall stack-rerandomized.
- **PAX_REFCOUNT** — `bnxt` refcount, per-VNIC refcount, per-ring refcount, per-VF refcount, ULP-slot refcount, HWRM-ctx refcount use saturating refcounts; underflow at remove is a hard panic.
- **PAX_MEMORY_SANITIZE** — HWRM DMA buffers `kfree_sensitive` (carry MAC addresses, RSS keys, CFA flow keys, RoCE QP context). Per-VNIC RSS context cleared on free. Per-VF context cleared on disable_sriov (carries trust state, MAC, VLAN — leak across VF reassignment is a tenant break).
- **PAX_UDEREF** — SMAP/SMEP on every user-facing entry.
- **PAX_RAP / kCFI** — `bnxt_netdev_ops`, `bnxt_ethtool_ops`, HWRM async event handler, ULP ops, TC HW-offload action dispatcher, devlink reporter callbacks all kCFI-signed.
- **GRKERNSEC_HIDESYM** — debugfs (`/sys/kernel/debug/bnxt/<bdf>/*`) dumps include ring pointers, HWRM buffer addresses; non-root reads scrub. Coredump regions also gated.
- **GRKERNSEC_DMESG** — bnxt log lines that reveal SR-IOV topology, NPAR partition layout, ULP-attached RDMA QP counts restricted from non-root.
- **CAP_NET_ADMIN** required on every state-changing op: ethtool MAC/RSS/coalesce/ring/NVM/flow-rule, devlink param set, SR-IOV enable, switchdev mode, TC HW-offload (gated through netdev permissions), VF config (MAC/VLAN/trust/rate/spoofchk).
- **HWRM sentinel structural** — Rookery's `HwrmReq<'_, T>` RAII guard makes use-after-free impossible at compile time. The upstream sentinel becomes a runtime backup; the type system is the primary defense.
- **HWRM opcode allowlist per-VF** — VFs are restricted to a subset of HWRM opcodes; opcodes that affect PF or other VFs are refused at the HWRM dispatch layer. Closes the historical "VF issues PF-level HWRM and reconfigures the whole NIC" CVE class.
- **HWRM response length validation** — every response is validated against the expected length for the requested opcode + FW version before parsing. A compromised/buggy FW returning a short response cannot trigger an OOB read in the parser.
- **HSI ABI version negotiated strictly** — driver only accepts HWRM_VER_GET responses with a version in the supported range; older FW versions logged + refused; newer FW versions get warning + best-effort.
- **CFA flow-rule install cap per representor** — per-representor TC rule install rate-limited + count-capped. Closes the "hostile container orchestrator OOMs CFA table by installing rules" class.
- **SR-IOV VF resource cap enforced** — PF refuses to allocate >max_vfs * per_vf_resource_quota; closes a resource-exhaustion path where allocating extra VFs without releasing prior ones could over-commit the PF.
- **Per-VF tenant-isolation metadata strict** — switchdev mode requires CFA-side per-VF tagging; without it, refuse switchdev mode for that PF. Parallels mlx5's match_metadata requirement.
- **Coredump access gated** — devlink-region read for the coredump region requires CAP_SYS_ADMIN; the coredump may contain in-flight packet data + RoCE key material.
- **NVM access gated** — firmware NVM read/write via devlink-region or ethtool eeprom-write requires CAP_SYS_ADMIN + lockdown mode allows. A compromised root with lockdown can't flash arbitrary firmware (signature-check required, see next bullet).
- **FW image signature check mandatory** — `HWRM_FW_INSTALL` / NVM-write paths verify the FW image's Broadcom-signed manifest before issuing the install HWRM. Unsigned images refused even with CAP_SYS_ADMIN.
- **Recovery rate-limit** — FW-reset path rate-limited; >3 resets in 60s → permanent down.
- **PTP CAP_SYS_TIME** — clock-adjust ops require CAP_SYS_TIME.
- **Debugfs CAP_SYS_ADMIN** — debugfs writes (HWRM-replay, ring-dump, reset-trigger) all require CAP_SYS_ADMIN.
- **XDP-attach scoped** — XDP-prog attach requires CAP_NET_ADMIN over the netdev (cgroup-scoped containers can be confined).

Per-doc rationale: bnxt is the highest-line-count driver Rookery documents. The HWRM transport is the single biggest attack surface — a ~10K-line ABI with hundreds of opcodes, multiple FW versions in production, and FW-version-dependent struct layouts. Historical CVEs concentrate in the HWRM parsing layer (CVE-2024-35814 nested-NIC overflow, CVE-2021-3635 ethtool OOB). The Rust translation's typed HWRM request/response (with RAII guards replacing the sentinel pattern) closes the largest historical bug class structurally. The HWRM opcode allowlist per-VF closes the SR-IOV-PF-escape class (a well-known historical hot spot). Recovery rate-limit + per-rule cap + per-VF resource cap close DoS classes from hostile orchestrators / compromised tenants. With Azure HC-series + bare-metal cloud running bnxt at scale, the cloud-perimeter NIC trust model partially applies (the host-side firmware is more trusted on bare-metal than on virtualized cloud, but post-FW-update the boundary shifts; signature-check on FW install is the mitigation).

## Open Questions

- [ ] Q1: Should the HWRM transport be auto-generated from `hsi.h` or hand-translated? Auto-gen via build.rs preserves spec-fidelity and gets free upstream updates; hand-translated is cleaner but lags. Recommendation: build.rs + hand-curated overrides for the rare struct that the generator can't express idiomatically.
- [ ] Q2: bnxt's 17500-line single file — should the Rust translation preserve the monolithic structure or split aggressively? Recommendation: split by concern (lifecycle, NAPI, ring-mgmt, irq, recovery) but keep them in one crate; cross-cutting `&BnxtPriv` references are unavoidable.
- [ ] Q3: NPAR (NIC Partitioning) — multi-PF per port is configured at FW level (NVM setting). Driver-side: each PF is independent. Question: should devlink expose NPAR config knobs (currently only some)? Or is NVM-only fine?
- [ ] Q4: bnxt_re ULP — separate kmod upstream, but Rookery has one binary. Same question as MANA mana_ib: aux-bus pattern preserved or directly embedded?
- [ ] Q5: TC HW-offload with conntrack — the bnxt CT path interacts with the kernel CT subsystem via NETFILTER notifiers. Robust handling of zone reassignment, CT entry timeout, and per-zone offload limit needs careful design.
- [ ] Q6: SmartNIC variants (Stingray on BCM57500) — different driver path. Should this Tier-3 cover Stingray or punt to a future iteration? Recommendation: punt — SmartNIC has very different lifecycle (full Linux running on the SmartNIC ARM cores) and warrants its own doc.

## Verification

- **Kani SAFETY**: prove `HwrmReq<'_, T>` cannot be used after drop (typestate). Prove sentinel check would catch any UAF that bypasses the type system. Prove VNIC-RSS-context index validation cannot be bypassed in the fast path.
- **TLA+**: model the FW-reset state machine with concurrent HWRM cmds + async events + netdev ifdown. Check no leaked rings, no leaked HWRM ctxs, no leaked VNICs after any sequence.
- **Verus**: functional spec of `bnxt_tc_setup_flower` → CFA-rule lowering: for every TC-flower spec accepted, the produced CFA rule has match-mask equivalent to the original and action-list equivalent (modulo documented HW-incompatibility rejections).
- **Kani+Verus**: invariant that every allocated VNIC is freed at netdev close; every allocated CFA flow is freed at TC rule del; every VF resource is released at disable_sriov.
- **Integration**: dual-100G iperf3 line rate. SR-IOV 64-VF stress (allocate, configure each, traffic on each, release in random order). Switchdev + TC HW-offload smoke (10K flow rules installed, traffic per rule, rules deleted). RoCE smoke via bnxt_re. FW reset injection. AER injection. NPAR with 4 PFs per port stress.
- **Fuzz**: HWRM response fuzzer (mutate length, status, payload per opcode); should never produce oops. Async event payload fuzzer. CFA flow-rule fuzzer (random TC-flower spec, validate parser); rejected specs should produce -EINVAL without partial install.
- **Penetration**: with an unprivileged container holding NET_ADMIN scoped to one VF, attempt to issue PF-level HWRM (e.g. HWRM_FUNC_CFG targeting a different VF); must be refused. SR-IOV passthrough to 16 KVM guests with isolation test.
