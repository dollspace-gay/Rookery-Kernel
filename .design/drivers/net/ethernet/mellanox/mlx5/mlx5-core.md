## Summary

The mlx5 core driver — the lowest layer of the Mellanox/NVIDIA ConnectX (4/5/6/7/8) and BlueField DPU host-side driver stack. Every consumer of the hardware — `mlx5e` (Ethernet netdev), `mlx5_ib` (InfiniBand/RoCE verbs), VDPA (`mlx5_vdpa`), TLS/IPsec accel, vfio-pci-mlx5, eSwitch, sub-functions (SF) — talks to firmware through this core. It owns: PCI device lifecycle (probe/remove/SR-IOV/AER/sriov/FLR), the firmware command interface (a 32-slot DMA-resident mailbox queue with both polling and event-driven completion modes), HCA capability discovery, the event-queue (EQ) machinery that demultiplexes async events from the device, the completion-queue (CQ) resource allocator that downstream consumers register through, the health buffer that surfaces firmware errors + auto-recovery, devlink integration, and the host-page service that satisfies firmware ICM (Internal Context Memory) allocations.

This Tier-3 covers the ~9000 lines of `mlx5/core/{main,cmd,fw,fw_reset,eq,cq,health,dev,alloc,devlink,pagealloc}.c`. The en/ subdriver is covered separately in `mellanox-mlx5-en.md`; this doc covers what en/ (and ib/vdpa/etc.) sits on top of.

## Rust translation posture

mlx5 core is one of the highest-value Rust translation targets in the kernel: it is widely deployed, heavily exercised, has had multiple historical CVEs in its command-mailbox + page-alloc paths, and its data structures are amenable to typed-state-machine modelling. Rookery's translation target:

- `mlx5_core_dev` → `Arc<MlxCoreDev>` with typed phase markers (`Probed<Pci> → Caps<HcaCaps> → Initialized<Hca> → Up`). Phase transitions go through `&mut self` so caller code cannot accidentally invoke `enable_hca`-needing operations on a `Probed<Pci>`.
- `mlx5_cmd` mailbox slots → fixed-capacity `[Cell<CmdSlot>; 32]` with a separate `Bitmap32` of in-flight slots; no `kmalloc` per command. Each `CmdSlot` owns its DMA-mapped pages (`DmaPages<Bidirectional>`), which `Drop` releases.
- Firmware `MLX5_CMD_OP_*` opcodes → typed methods on `FwCmd<'_>`: `fw.query_hca_cap(...) -> Result<HcaCaps, FwError>` rather than `mlx5_cmd_exec_inout(dev, query_hca_cap, in, out)` with `MLX5_SET`/`MLX5_GET` macro accessors.
- `mlx5_ifc.h` bit-packed structures (autogenerated from the firmware ABI) → `bilge`-or-equivalent typed bitfield structs with bounds-checked accessors. The macros `MLX5_GET8/16/32/64` get replaced by typed `cap.log_max_qp() -> u8`, etc.
- `mlx5_eq_table` interrupt vectors → `Vec<Arc<EventQ>>` with explicit ownership; affinity moves go through the irq-affinity-manager.
- Health buffer + recovery → tokio-style task; the watchdog uses `Notify` rather than the kernel `delayed_work`.

The grsec/PaX bullets below are *mandatory* — mlx5 is a network-facing, firmware-mediated, IOMMU-touching driver and historically a productive bug-hunting target.

## Compatibility contract

REQ-1: PCI device-id table matches every ConnectX-4 through ConnectX-8 PF + VF + BlueField host PF + SF representor — list-of-vendor-IDs frozen at the same set as upstream `mlx5_core_pci_table[]` so any device that loads upstream-mlx5 loads Rookery-mlx5.

REQ-2: HCA capabilities — `MLX5_CAP_GENERAL`, `MLX5_CAP_ETH`, `MLX5_CAP_FLOW_TABLE`, `MLX5_CAP_ESWITCH`, `MLX5_CAP_ROCE`, `MLX5_CAP_QOS`, `MLX5_CAP_VECTOR_CALC`, `MLX5_CAP_DEBUG`, `MLX5_CAP_ODP`, `MLX5_CAP_IPSEC`, `MLX5_CAP_MACSEC`, `MLX5_CAP_TLS`, `MLX5_CAP_VDPA_EMULATION`, `MLX5_CAP_DEV_MEM`, `MLX5_CAP_PSP`, `MLX5_CAP_CRYPTO`, `MLX5_CAP_ADV_VIRTUALIZATION`, `MLX5_CAP_FLOW_MATCH_SAMPLE_FIELD_SELECT_REGISTER_C`, etc. — all query-able through `mlx5_core_get_caps()` equivalent. Consumers (en/ib/vdpa) must see the same cap surface they see upstream.

REQ-3: Firmware command interface — 32-entry mailbox at the PCI BAR, polling and event-driven modes, command-mode toggle via `mlx5_cmd_use_events()` after async-EQ is up, completion via either polling (during init pre-EQ) or `cmd_event` IRQ. Command result statuses (`MLX5_CMD_DELIVERY_STAT_*`) translated to `-errno` per upstream `cmd_status_to_err()` mapping.

REQ-4: Mailbox structure ABI — `mlx5_ifc_mbox_in_bits` (opcode/uid/op_mod) + `mlx5_ifc_mbox_out_bits` (status/syndrome) at the head of every command. Opcode space `MLX5_CMD_OP_QUERY_HCA_CAP`..`MLX5_CMD_OP_*` enumerated exactly as `include/linux/mlx5/mlx5_ifc.h` (frozen against baseline; no extensions until upstream adds them).

REQ-5: Init-phase sequence: enable PCI → request BARs → set DMA mask 64 → cmd init → `enable_hca` → `query_pages(boot)` → serve boot pages → `init_hca` → `set_hca_cap` → `query_pages(init)` → serve init pages → `set_hca_defaults` → eq table init → async EQs (cmd/pages/async/error) → health poll start → devlink register. Upstream order frozen; any divergence is a compat break.

REQ-6: Host pages — firmware can request additional pages at any time via `MLX5_EVENT_TYPE_PAGE_REQUEST`. Driver maps + DMAs them and replies `MANAGE_PAGES(GIVE)`; on `RECLAIM` reverse. Page tracking via per-function xarray (`xa_store`/`xa_erase`); on driver unload, all outstanding pages must be reclaimed.

REQ-7: Health buffer — at fixed offset within the PCI BAR (`mlx5_core_dev->iseg->health`), polled at HEALTH_POLL_INTERVAL (~2s). Fatal-state detect (`MLX5_HEALTH_SYNDR_*`) triggers `mlx5_handle_bad_state`. Recovery via FLR (Function-Level Reset) or fw-activated reset, coordinated across PF/VF/SF via the FW-reset state machine in `fw_reset.c`.

REQ-8: Event queues — at least 4 async EQs: command-completion (`cmd_eq`), page-request (`pages_eq`), async-event (`async_eq`), error (`async_eq` doubles as carrier of `MLX5_EVENT_TYPE_INTERNAL_ERROR`). Plus one comp-EQ per data-path channel (created by en/ib via `mlx5_comp_eqn_get`).

REQ-9: SR-IOV — `pci_sriov_configure` callback enables N VFs. Each VF gets its own `mlx5_core_dev` instance (probe path runs for both PF and VF); per-VF cap is `MLX5_CAP_GENERAL.vhca_resource_manager`. eSwitch (separate doc) provides the representor netdev on the PF side.

REQ-10: AER handlers — `mlx5_pci_err_detected` (drains in-flight cmds, stops health poll), `mlx5_pci_slot_reset` (re-enable PCI, re-init), `mlx5_pci_resume` (full load). Mandatory for cloud deployment because PCIe AER is the failure-detect path.

REQ-11: Devlink params — `enable_roce`, `io_eq_size`, `event_eq_size`, `max_macs`, `disable_netdev_queues`, `cmode_runtime`/`cmode_driverinit` discipline preserved. Devlink region snapshot for cr-dump diagnostic capture.

REQ-12: Module unload — `mlx5_unload_one()` must reverse every step of load; all CQs/QPs/SRQs/EQs destroyed; all pages reclaimed; cmd cleanup; PCI disable. Refcount on `mlx5_core_dev` must reach zero before `kfree`.

## Architecture

**Phase-typed device lifecycle.** Rookery models the multi-phase init of `mlx5_function_setup` → `init_hca` → `set_hca_defaults` → `load` as typed transitions on `MlxCoreDev<Phase>` where `Phase` advances through `Probed<Pci>` → `Cmd<Ready>` → `Caps<Queried>` → `Hca<Initialized>` → `Loaded`. Each phase exposes only the methods valid for it. `enable_hca` consumes a `Cmd<Ready>` and yields a `Caps<Queried>`; `init_hca` consumes a `Caps<Queried>` and yields an `Hca<Initialized>`. The PCI error/unload paths take any phase by `Drop` impl and emit the correct teardown command sequence based on which phases ran.

**Firmware command interface.** The mailbox is a 32-slot array of `CmdSlot { in_dma: DmaPages<ToDevice>, out_dma: DmaPages<FromDevice>, lay: Lay /* 64B-aligned descriptor */, completion: Notify, status: AtomicCell<CmdStatus> }`. The slot allocator is a single `AtomicU32` bitmap; reservation via `fetch_and(!free_bit)` with an `arc_swap` of the slot-claim. The completion path differs by phase: during init we **poll** (the EQ isn't up yet); once `cmd_use_events()` runs, completions arrive on the cmd-EQ IRQ and wake `slot.completion`. Both paths converge in `FwCmd::exec_sync` returning `Result<CmdOut, FwError>` where `FwError` distinguishes `Delivery(_)` (mailbox transport) from `Operation { status: u8, syndrome: u32, opcode: u16 }` (firmware logical error).

**Page service.** Firmware doesn't allocate its own ICM (Internal Context Memory) — it asks the driver for host pages via `MLX5_EVENT_TYPE_PAGE_REQUEST` on the pages-EQ. The handler reads `dev->required_pages`, allocates that many `Page`s (each DMA-mapped, recorded in a per-function `BTreeMap<u64 /*dma*/, Page>`), and issues `MANAGE_PAGES(GIVE)` via the cmd interface. On `RECLAIM`, the reverse: pop pages off the map, unmap, free. Page accounting is reconciled at unload — leaked pages are a fatal driver bug.

**Health buffer + recovery.** A 64-byte structure mapped at the head of the PCI BAR; firmware writes a synd byte + ASN-blame info when it hits an internal fault. The `Health` task wakes every ~2s, reads the buffer, and if the synd byte is nonzero, enters the recovery state machine. Recovery options (in order): try fw-activated reset (preserves SR-IOV topology), fall back to PCIe FLR (resets the function). All in-flight commands get `-EIO` returned; downstream subsystems (en/ib) must reattach after recovery via the `mlx5_blocking_notifier_call_chain` mechanism.

**EQ demultiplex.** Each EQ entry is a `mlx5_eqe` with `type/sub_type/data[...]`. The async-EQ handler dispatches by `type` to per-subsystem listeners registered via `mlx5_eq_notifier_register`. Cmd-EQ entries carry the slot number — the handler matches the slot index to the in-flight slot table and wakes the waiter. Comp-EQs do not flow through this core dispatch — they go directly into NAPI in en/ib drivers.

## Hardening

- All cmd inputs zero-initialized before fill (`memset(in, 0, sizeof(in))` upstream — Rookery uses `MaybeUninit::zeroed().assume_init()` for ABI-shaped outputs only after a successful FW exec).
- DMA-map every command's in/out buffers separately; never share. Use `DMA_FROM_DEVICE`/`DMA_TO_DEVICE` directionality correctly so IOMMU can enforce.
- Health-buffer reads use `READ_ONCE`-equivalent volatile semantics; firmware can update mid-poll.
- Page-request handler caps `required_pages` at a per-function maximum (against a hostile firmware demanding 2^32 pages); upstream cap is `MAX_RECLAIM_NPAGES` / device-mem-cap.
- Cmd-status to `-errno` mapping never returns `0` from a non-success status (a productive historical bug class: misclassifying `OK_WITH_BAD_PARAMS` as success).
- `mlx5_core_dev` refcount is checked at `kfree`; any leak fails loud rather than freeing while held.
- All `MLX5_GET/SET` macro accesses go through bounds-checked typed accessors so an attacker-controlled `cap_type` cannot read past the cap buffer.

## Grsecurity/PaX-style Reinforcement

This driver inherits the standard PaX/Grsecurity surface and reinforces it with:

- **PAX_USERCOPY** — every cmd mailbox in/out buffer is a fixed-size, kernel-managed DMA buffer; no user pointers cross the firmware boundary. Devlink param writes that copy from user (`devlink_param_value_set`) go through bounded `copy_from_user_with_bounds`.
- **PAX_KERNEXEC** — `mlx5_cmd_ops` vtables, eqe notifier chains, devlink-op tables, AER callback tables are all `const` after init and live in W^X memory.
- **PAX_RANDKSTACK** — every cmd-exec entry path is wrapped in the per-syscall stack-rerandomization wrapper; the mailbox slot path does not touch user stack but the issuer (ioctl/devlink/sysfs) does.
- **PAX_REFCOUNT** — `mlx5_core_dev->refcount`, per-CQ/EQ refcounts, page-map entry refcounts use saturating-overflow refcount type; underflow at `mlx5_uninit_one` is a hard panic.
- **PAX_MEMORY_SANITIZE** — DMA-coherent buffers used for cmd in/out are `kfree_sensitive`-equivalent: zeroed on free. Firmware command payloads can include keys (TLS-DEK, IPsec SA-key, PSP key, MACsec key) — these MUST sanitize.
- **PAX_UDEREF** — SMAP/SMEP enforced on every cmd issuer path; the cmd interface itself does not access user memory but its callers do (ioctl/devlink/sysfs/ethtool).
- **PAX_RAP / kCFI** — `mlx5_cmd_status_str`, `mlx5_eq_notifier`, `mlx5_blocking_notifier_call_chain`, and the per-cap query-handler tables are kCFI-signed; indirect calls through them are signature-checked.
- **GRKERNSEC_HIDESYM** — `dmesg` traces from `mlx5_core_dbg` / `mlx5_core_warn` are scrubbed of kernel addresses and IRQ-vector numbers when read by non-root; the rich debug print of QP/CQ/EQ pointers stays gated behind `CAP_SYS_ADMIN`.
- **GRKERNSEC_DMESG** — firmware-tracer ringbuffer (`diag/fw_tracer.c`) restricted to root; can carry guest tenant addresses on a BlueField DPU host.
- **CAP_NET_ADMIN** on every devlink/devlink-port/devlink-rate write; ethtool ops that change MAC, MTU, FEC, queues all gated.
- **Firmware signature mandatory** — `flash_component` (devlink firmware update) refuses unsigned blobs; signature check uses the Mellanox FW signing root anchored at kernel build time.
- **Firmware-reset coordination strict** — `fw_reset.c` state machine refuses any reset while a sibling PF/VF has un-quiesced traffic, preventing a malicious tenant on one VF from forcing a reset of the whole HCA.
- **AER → recovery rate-limit** — if `pci_err_detected` fires more than 3x in 60s, the driver gives up and leaves the device offline rather than infinite-retry recovery.
- **Cmd-opcode allowlist per-uid** — when a user-space verb path (RDMA `mlx5_ib` for an unprivileged process) lands a command, only the opcode subset that uid is allowed to issue is accepted. Privileged opcodes (CREATE_GENERAL_OBJECT subset, MODIFY_HCA_VPORT_CONTEXT, INIT_HCA) require CAP_NET_ADMIN.
- **Page-request DoS cap** — firmware's `MLX5_EVENT_TYPE_PAGE_REQUEST` is rate-limited to N pages/second per function; a runaway-FW DoS hits the cap and trips health recovery.
- **VHCA-id randomization** — when the driver allocates the software-VHCA-id (used to differentiate SR-IOV functions in some firmware ops), the allocation order is randomized (LATENT_ENTROPY) to make cross-VF probing harder.

Per-doc rationale: mlx5 sits at the intersection of three high-risk surfaces — PCIe DMA (IOMMU-bypass if misconfigured), firmware command interface (historical CVE source — CVE-2018-1077, CVE-2022-3635 mailbox-handling bugs), and SR-IOV / sub-functions (one VF compromising the PF). The reinforcement above is not optional; each item closes a class of bug that has shipped before on this driver. Rookery's typed-state-machine model is the structural defense — the bullet list above is the runtime/kCFI defense layered on top.

## Open Questions

- [ ] Q1: Should Rookery preserve upstream's `prof_sel` module-param (selects between 4 baked-in HCA profiles) or move to devlink-only configuration? Upstream is moving toward devlink; Rookery could skip the legacy param.
- [ ] Q2: Cmd-EQ vs cmd-polling — upstream toggles to events at `cmd_use_events()` time. Rookery could stay polling for the first N cmds even after EQs are up, simplifying the state machine; cost is ~µs latency on init-time cmds (acceptable).
- [ ] Q3: Page-service backing — upstream uses `alloc_pages_node` with NUMA hinting. Rookery's allocator currently doesn't expose per-node hints; a node-aware allocator path is a prerequisite for matching upstream perf on dual-socket hosts.
- [ ] Q4: FW signature trust anchor — upstream relies on `request_firmware()` + LSM `kernel_load_data` hook. Rookery's LSM stack must hook the same point; design needs the LSM hook before this driver ships.
- [ ] Q5: Devlink region snapshot format — upstream uses an opaque blob. Rookery could replace with a typed CBOR/JSON dump; trade-off is compatibility with Mellanox-provided cr-dump tools.

## Verification

- **Kani SAFETY**: prove `CmdQ::reserve_slot` cannot return a slot index already in use (bitmap-allocator invariant). Prove `PageSvc::serve` page-count cannot underflow on RECLAIM (saturating subtract).
- **TLA+**: model the FW-reset state machine across PF/VF/SF roles; check liveness (every reset request eventually transitions to `RESET_DONE` or `RESET_FAILED`, no deadlock) and safety (no two functions can be in `RESETTING` simultaneously on the same HCA).
- **Verus**: functional spec of `FwCmd::exec_sync` — given a well-formed in-buffer and a non-failing firmware oracle, returns the expected out-buffer matching the `MLX5_CMD_OP_*` ABI.
- **Kani+Verus**: invariant that every page given to firmware (`MANAGE_PAGES(GIVE)`) is recorded in the page-map and every page-map entry corresponds to a `MANAGE_PAGES(GIVE)` that has not yet been reclaimed — i.e., no leaks, no double-frees.
- **Integration**: dual-host RoCE rdma-ping (ConnectX-6), TCP iperf3 100 Gb/s sustained for 1h, FLR-storm test (1000 FLRs/min for 10min) with no kernel splat, SR-IOV passthrough to QEMU/KVM guest with vfio-pci.
- **Fuzz**: command-mailbox fuzzer (mutate out-buffer the firmware returns) against the cmd-status decode + cap-query path; should never produce a kernel oops.
