## Summary

`inotify_rm_watch(2)` detaches the watch identified by `wd` from the inotify instance referred to by `fd`. The kernel atomically: (a) removes the mark from the group's IDR by `wd`, (b) detaches the mark from its target inode via `fsnotify_destroy_mark`, (c) decrements `UCOUNT_INOTIFY_WATCHES` for the owning uid, (d) enqueues a final `IN_IGNORED` event (`wd == removed_wd`, `mask == IN_IGNORED`) so that a reader sees the death of the watch. The fd remains open and other `wd`s in the group are unaffected. Critical for: build systems removing transient watches, daemons re-scoping observation, container runtimes shutting down per-container watch sets, garbage-collection of `IN_ONESHOT` consumers.

This Tier-5 covers the userspace ABI of syscall 255.

## Signature

```c
int inotify_rm_watch(int fd, int wd);
```

Rust ABI shim:

```rust
pub fn sys_inotify_rm_watch(fd: i32, wd: i32) -> isize;
```

Syscall number: **255**.

## Parameters

| Param | Type | Direction | Purpose |
|---|---|---|---|
| `fd` | `i32` | IN | inotify instance from `inotify_init1(2)` |
| `wd` | `i32` | IN | watch descriptor returned by `inotify_add_watch(2)` |

## Return

- **Success**: `0`.
- **Failure**: `-1` and `errno`; Rust internal returns negated errno.

## Errors

| errno | Trigger |
|---|---|
| `EBADF` | `fd` not an inotify fd, or `fd` invalid |
| `EINVAL` | `wd` invalid for this group, or `fd` not an inotify fd |

## ABI surface

- No flag arguments.
- `wd` integer space: per-group, allocated cyclically via `idr_alloc_cyclic` from 1 upward.
- `IN_IGNORED` event constant (`0x00008000`): generated as the final event for the removed `wd`.

## Compatibility contract

REQ-1: `fd` validation:
- Must be an inotify-fd (file_operations match); else `-EINVAL` (note: not `-EBADF` if fd exists but wrong type — kernel reports `-EINVAL`).
- `fd` invalid (closed / never valid) ⟹ `-EBADF`.

REQ-2: `wd` validation:
- Must be a currently-allocated watch descriptor in this group; else `-EINVAL`.
- `wd <= 0` ⟹ `-EINVAL`.

REQ-3: Atomic remove sequence:
- Acquire group's mark-write lock.
- `idr_find(wd)`; if NULL ⟹ `-EINVAL`.
- Remove from IDR (`idr_remove`).
- Drop write lock.
- `fsnotify_destroy_mark(mark, group)` — detaches from inode list, triggers `freeing_mark` callback.

REQ-4: `IN_IGNORED` event enqueue:
- Generated by `inotify_freeing_mark` callback.
- `event.wd = wd`, `event.mask = IN_IGNORED`, `event.cookie = 0`, `event.len = 0`.
- Enqueued onto group's event queue (subject to `max_queued_events`).
- Delivered to reader on next `read(2)`.

REQ-5: ucount accounting:
- `dec_ucount(UCOUNT_INOTIFY_WATCHES)` when mark is freed.

REQ-6: Idempotency:
- Calling twice with same `wd`: second call returns `-EINVAL`.

REQ-7: Concurrency:
- Concurrent `inotify_add_watch(same path, MASK_ADD)` may have allocated a fresh `wd` before this rm completes; the rm targets only the named `wd`.
- Concurrent `read(2)` may see events for the watched inode that were enqueued before this rm; ordering: pre-rm events, then `IN_IGNORED`, then no further events.

REQ-8: Implicit removal:
- `IN_ONESHOT` watches self-remove on first event delivery; subsequent `inotify_rm_watch(wd)` returns `-EINVAL`.
- Inode removal (file deleted, filesystem unmounted) triggers implicit watch removal with `IN_IGNORED`.

REQ-9: `close(fd)`:
- Implicit rm of all watches in the group; each watch generates `IN_IGNORED` but the queue is being drained, so readers may not observe them.

REQ-10: User-namespace scope:
- Inherited from the inotify-fd's group; rm dec's ucount in the group's user_ns.

## Acceptance Criteria

- [ ] AC-1: After `inotify_add_watch(fd, p, IN_MODIFY)` returning `wd`, `inotify_rm_watch(fd, wd) == 0`.
- [ ] AC-2: After successful rm, `read(fd, buf, ...)` eventually delivers `{wd, IN_IGNORED, 0, 0}`.
- [ ] AC-3: `inotify_rm_watch(fd, wd)` twice → second call `-EINVAL`.
- [ ] AC-4: `inotify_rm_watch(fd, 0)` → `-EINVAL`.
- [ ] AC-5: `inotify_rm_watch(fd, -1)` → `-EINVAL`.
- [ ] AC-6: `inotify_rm_watch(fd, INT_MAX)` (unallocated) → `-EINVAL`.
- [ ] AC-7: Bad fd (closed) → `-EBADF`.
- [ ] AC-8: Non-inotify fd (regular file) → `-EINVAL`.
- [ ] AC-9: After rm, modifications to watched inode do not generate further events.
- [ ] AC-10: After rm, `UCOUNT_INOTIFY_WATCHES` for caller's uid is decremented by 1.
- [ ] AC-11: `IN_ONESHOT` watch: after first event delivery, `inotify_rm_watch(wd)` returns `-EINVAL`.
- [ ] AC-12: `close(fd)` after add_watch but no rm: ucount decremented as part of release.
- [ ] AC-13: Concurrent rm on two distinct `wd`s in same group: both succeed; no event-queue corruption.

## Architecture

Rookery surface in `kernel/notify/inotify/syscall.rs`:

`Inotify::inotify_rm_watch(fd, wd) -> isize`:
1. if wd <= 0: return -EINVAL.
2. let file = fdget(fd).ok_or(-EBADF)?;
3. let group = file.private_data.downcast::<InotifyGroup>().ok_or(-EINVAL)?;
4. let mut marks = group.watches.write();
5. let mark = marks.remove_by_wd(wd).ok_or(-EINVAL)?;
6. drop(marks);
7. fsnotify_destroy_mark(&mark.fsn_mark, group);
8. /* fsnotify_destroy_mark triggers Inotify::freeing_mark via fsnotify_ops */
9. return 0;

`Inotify::freeing_mark(mark, group)` (fsnotify callback):
1. let event = InotifyEvent { wd: mark.wd, mask: IN_IGNORED, cookie: 0, name: None };
2. /* Enqueue under queue lock; respect max_queued_events */
3. group.queue.lock().push_back_bounded(event, group.max_events);
4. group.wq.wake_all_with(POLLIN);
5. /* ucount dec happens in inotify_mark_free below */

`Inotify::inotify_mark_free(mark)`:
1. dec_ucount(UCOUNT_INOTIFY_WATCHES).
2. drop(mark) — Arc refcount goes to zero; FsnotifyMark destructor runs.

## Hardening

(Inherits row-1 features from `uapi/00-overview.md` § Hardening.)

inotify_rm_watch reinforcement:

- **Strict wd validity** — defense against per-stale-wd use confusion.
- **IDR remove before mark destroy** — defense against per-double-free via concurrent rm.
- **`IN_IGNORED` enqueue under queue lock** — defense against per-event-loss / per-double-emit.
- **ucount dec paired with mark free** — defense against per-quota imbalance.
- **`inotify_rm_watch` idempotent only via -EINVAL** — never returns success on stale wd.

## Grsecurity/PaX-style Reinforcement

- **PaX UDEREF inherited** — the syscall takes only scalars; no user pointers. UDEREF discipline applies at the generic syscall entry path.
- **GRKERNSEC_HIDESYM on `IN_IGNORED` event timing side-channel** — the timing between `inotify_rm_watch` invocation and `IN_IGNORED` enqueue can be measurable; under hardened policy, queue the IN_IGNORED synchronously inside the syscall to eliminate the timing window that lets a coresident observer infer rm activity on a shared group.
- **Audit log on rm-after-fork** — if `wd` was added by parent pre-fork and child rm's it after exec, log at `LOGLEVEL_INFO`. Forensically interesting for tracking watch-handoff across privilege boundaries.
- **ucount dec strict pairing audit** — Kani invariant `add → wd → rm → dec`; runtime check that `UCOUNT_INOTIFY_WATCHES` matches the IDR cardinality across the group's mark set at every rm; mismatch logs `BUG_ON` under hardened build.
- **GRKERNSEC_CHROOT_FINDTASK-style fence** — `inotify_rm_watch` is permitted across chroot boundaries (the fd is the gate, not the watched path); however, when the watched inode's mount has since been moved across a chroot boundary the rm proceeds normally — there is no escape path here. Audit doctrine merely logs the cross-boundary case.
- **PAX_RANDKSTACK on rm entry** — randomize kernel stack on each rm; rm is the deallocation path and a useful target for kstack-layout inference paired with add.
- **Refuse rm during group teardown** — if group is in `INOTIFY_GROUP_RELEASING` state, refuse with `-EINVAL` rather than racing the release path. The release will free all marks itself; an in-flight rm would race the IDR teardown.
- **rate-limit `IN_IGNORED` flood** — bulk rm of N watches generates N IN_IGNORED events; under hardened policy, if a single syscall sequence rm's more than `64` watches within `100 ms`, batch the IN_IGNORED into a single event with `cookie = N`. Defense against per-event-storm queue exhaustion.
- **CAP_LINUX_IMMUTABLE not required, but logged** — rm of a watch on an `S_IMMUTABLE` inode is permitted (the immutability applies to data writes, not to inotify marks), but the combination is logged under hardened policy.
- **No CAP_SYS_ADMIN required** — baseline primitive. Watch ownership gates the rm: only the group holding the wd can rm it; cross-process rm is impossible because `fd` is per-process.
- **`fsnotify_destroy_mark` ordering invariant** — IDR remove must precede mark destroy; mark destroy must precede ucount dec; ucount dec must precede group-release completion. Violation triggers `WARN_ONCE` under hardened build.
- **TOCTOU on concurrent add/rm** — concurrent `inotify_add_watch(MASK_ADD)` on the same inode while rm is in flight: rm targets the named `wd`; the add either sees rm's removal and allocates a fresh `wd`, or merges into the still-live old `wd`. The grsec doctrine logs this race for forensic correlation under heavy fuzzing.
